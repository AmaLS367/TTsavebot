# Техническое задание (ТЗ): Telegram-бот для скачивания видео (TikTok / Instagram)

## 1. Описание проекта
Telegram-бот, предоставляющий администраторам возможность автоматизированного и бесплатного скачивания видео из TikTok и Instagram без водяных знаков (watermarks) по прямой ссылке. Бот будет являться частью конвейера (Этап 2 основного проекта) для массового сбора исходников (девушек-доноров) под FaceFusion.

## 2. Функциональные требования
- **Прием ссылок:** Бот должен распознавать ссылки на TikTok (`tiktok.com`, `vm.tiktok.com`) и Instagram (`instagram.com/reel`, `instagram.com/p`).
- **Скачивание видео:** 
  - Загрузка видео из TikTok без водяного знака.
  - Загрузка Reels из Instagram в максимальном качестве.
- **Отправка файлов:** Бот должен отправлять скачанный видеофайл в чат администратору (пользователю).
- **Ограничения размера:** Учет лимита Telegram Bot API на отправку файлов (до 50 МБ для обычных ботов). В 99% случаев ролики Reels/TikTok весят меньше.
- **Права доступа (Auth):** Ограничение доступа к боту (White-list ID пользователей), чтобы предотвратить злоупотребление ресурсами сервера сторонними людьми.
- **Логирование:** Сохранение истории скачиваний (ссылка, пользователь, статус) для аналитики.

## 3. Нефункциональные требования
- **Язык программирования:** Python 3.10+
- **Фреймворк:** `aiogram 3.x` (современный асинхронный фреймворк).
- **Архитектура:** Clean Architecture (Чистая архитектура).
- **Надежность:** Обработка ошибок (недействительные ссылки, удаленные видео, блокировки от соцсетей, приватные аккаунты) с понятным уведомлением.
- **Асинхронность:** Скачивание видео и работа с диском не должны блокировать `event loop` бота. Применение `asyncio` или пула потоков.

## 4. Архитектура проекта (Clean Architecture)

Проект должен быть разделен на независимые слои (Layers) для обеспечения тестируемости, независимости от фреймворков и простоты замены компонентов (например, замена API библиотеки для скачивания при её поломке).

### 4.1. Domain Layer (Слой Сущностей)
Самый внутренний слой, содержащий корпоративную или бизнес-логику высокого уровня. Не зависит от библиотек.
* **Entities:** `User`, `VideoRequest`, `DownloadedVideo`.
* **Value Objects:** `URL`, `PlatformType` (TikTok/Instagram).

### 4.2. Application Layer (Слой Use Cases / Interactors)
Содержит бизнес-логику самого приложения. Зависит только от слоя Domain.
* **Интерфейсы (Ports):** 
  * `IVideoDownloaderService` (интерфейс для работы с парсерами).
  * `IAccessRepository` (работа с БД пользователей/авторизация).
  * `IFileStorage` (интерфейс работы с файловой системой).
* **Use Cases:**
  * `DownloadVideoUseCase(url, user_id)`: Оркестрирует весь процесс. Проверяет платформу по URL, обращается к `IVideoDownloaderService`, возвращает сущность видео.
  * `CheckAccessUseCase(user_id)`: Проверяет наличие прав.

### 4.3. Infrastructure Layer (Слой Инфраструктуры)
Реализация интерфейсов (Ports) из Application Layer. Здесь происходит работа с внешним миром.
* **Скачивание (Adapters):**
  * Реализация `IVideoDownloaderService` через `yt-dlp` (наиболее надежный консольный инструмент для загрузки видео с соцсетей, поддерживающий удаление watermark в TikTok и парсинг Inst).
  * Альтернативные реализации через публичные API (например, Cobalt API или веб-скрейперы).
* **База данных:** SQLite (через `aiosqlite`) для локального хранения White-list. Для данного масштаба PostgreSQL избыточен, но из-за интерфейсов его можно будет легко заменить.
* **Файловая система:** Локальное временное хранилище `os` / `aiofiles` для скачанных видеофайлов перед их отправкой в Telegram. Включает механизм очистки мусора.

### 4.4. Presentation Layer / Delivery (Слой Представления)
Точка входа запросов от Telegram.
* Использование `aiogram 3`.
* Обработчики сообщений (Handlers) / Routers.
* Парсинг входящего текста. Если текст = ссылка, вызов Use Case.
* Middlewares (например, `WhiteListMiddleware`), блокирующие запросы до попадания в Handler.

### 4.5. Структура директорий (Пример)
```text
video_bot/
├── core/                   # Domain & Application Layers
│   ├── entities/           
│   ├── use_cases/          
│   └── interfaces/         # Порты (IVideoDownloaderService и др.)
├── infrastructure/         # Infrastructure Layer
│   ├── database/           # SQLite (CRUD операции)
│   ├── downloaders/        # ytdlp_adapter.py , cobalt_adapter.py
│   └── file_storage.py     # Логика работы с /tmp_downloads
├── presentation/           # Presentation Layer (Telegram)
│   ├── handlers/           # message_handlers.py
│   ├── middlewares/        # auth_middleware.py
│   └── tg_bot.py           # Инициализация aiogram
├── .env                    # Токены
├── config.py               # Загрузка env
├── main.py                 # Точка входа, DI Container (сборка зависимости)
├── requirements.txt
└── Dockerfile
```

## 5. Стек технологий
* **Язык Core:** Python 3.11+
* **Telegram Bot:** `aiogram 3`
* **Scraping / Скачивание:** `yt-dlp` (используется через `asyncio.create_subprocess_exec` для асинхронного вызова консольных команд без блокировки потока).
* **БД (Опционально):** `SQLite` (модуль `sqlite3` или ORM `SQLAlchemy 2.0`).
* **Контейнеризация:** `Docker`.

## 6. Алгоритм работы (Workflow)
1. Администратор отправляет ссылку боту.
2. `AuthMiddleware` проверяет наличие ID пользователя в белом списке. Если нет — игнорирует запрос (не тратит ресурсы).
3. Handler перехватывает ссылку, определяет валидность.
4. Handler отправляет сообщение `"⏳ Загрузка видео, пожалуйста подождите..."`.
5. Вызывается `DownloadVideoUseCase`.
6. Infrastructure Layer загружает видео с помощью `yt-dlp` в папку `downloads/` с уникальным ID (UUID).
7. Telegram API отправляет локальный видеофайл администратору (`FSInputFile`).
8. После успешной отправки файл гарантированно удаляется из файловой системы через блок `finally`, чтобы диск сервера не переполнялся.
9. Исходное сообщение "Загрузка..." удаляется или редактируется.

## 7. Обработка рисков
* **Блокировки от Instagram:** Instagram активно борется с парсингом. `yt-dlp` может потребовать передачи файлов `cookies.txt` (авторизация с фейкового аккаунта). В Layer Infrastructure это нужно предусмотреть параметром.
* **Переполнение диска:** Реализация обязательно должна подчищать скачанные файлы, даже если отправка в TG упала с ошибкой.
* **Зависание скачивания:** Установить Timeout на процесс скачивания (например, не более 60 секунд), чтобы бесконечно долгие загрузки не убивали память сервера.

## 8. План разработки (Roadmap бота)
1. **Подготовка базы:** Создание структуры папок Clean Architecture. Настройка `main.py` и `config.py`.
2. **Инфраструктура:** Написание и тестирование адаптера `yt-dlp` (в виде скрипта) для скачивания видео из TikTok/Inst без водяных знаков в консоли.
3. **Application:** Написание Use Case для объединения загрузчика и файловой системы.
4. **Бот:** Подключение `aiogram`. Написание хэндлеров на прием ссылок.
5. **Безопасность:** Реализация White-list через Middleware.
6. **Оптимизация:** Доработка удаления файлов, управление таймаутами.
7. **Деплой:** Написание `Dockerfile`, развертывание бота на сервере через `docker run`.
